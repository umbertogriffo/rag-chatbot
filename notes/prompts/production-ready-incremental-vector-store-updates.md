Plan: Production-Ready Incremental Vector Store Updates
Replace the current full-rebuild ingestion with document-level version tracking, a doc_id → chunk_ids mapping table, and an incremental diff-based pipeline. Currently, chunks have no version_hash, deletions are unimplemented (TODO in delete_document), and memory_builder.py always rebuilds everything.

Steps
Add compute_version_hash to id_generator.py. Create a function that SHA-256 hashes a document's full content (using existing normalize_text) to produce a stable version fingerprint. This is distinct from chunk-level deterministic IDs — it represents the whole source document.
Enrich chunk metadata with document_id and version_hash. In split_chunks (or a new helper wrapping it), propagate document_id and version_hash into every chunk's metadata dict. Update the upload endpoint to include version_hash alongside the existing document_id it already sets. Update memory_builder.py build_memory_index to derive a document_id from the source path and compute a version_hash per source document.
Create a persistent DocumentRegistry class (new file, e.g. chatbot/bot/memory/document_registry.py) backed by a JSON file (or SQLite for concurrency) stored alongside the vector store. It tracks {document_id: {source, version_hash, chunk_ids: [...]}}. Provide methods: get_all(), get(doc_id), upsert(doc_id, info), remove(doc_id), get_version_hash(doc_id).
Add delete_chunks_by_document_id to Chroma. Implement it using self.collection.delete(where={"document_id": doc_id}) for the Chroma-native path, and also accept an optional chunk_ids list for precise targeted deletion via self.collection.delete(ids=chunk_ids) — using the mapping table when available for efficiency. Have from_chunks return the list of generated chunk IDs so they can be recorded in the registry.
Refactor build_memory_index into an incremental pipeline in memory_builder.py. The new flow: (a) load source docs and compute each doc's version_hash, (b) diff against DocumentRegistry to find new/changed/deleted docs, (c) delete old chunks for changed/deleted docs (via registry's chunk_ids), (d) chunk and ingest only new/changed docs, (e) update the registry with new version_hash + chunk_ids. Keep a --full-rebuild flag for the initial build or forced re-index.
Implement delete_document endpoint's TODO in documents.py. Use the new delete_chunks_by_document_id on the Chroma index (using chunk IDs from the registry for precision), then remove the document from the DocumentRegistry. Also replace the in-memory _documents dict with the DocumentRegistry for persistence across restarts.

Further Considerations
Registry storage backend: JSON file is simplest but not safe under concurrent writes (e.g. simultaneous uploads). SQLite with WAL mode would handle concurrency for the nightly job + API writes. Recommend starting with SQLite.
Nightly job orchestration: The incremental build_memory_index can be invoked via a cron/systemd timer calling memory_builder.py. Should it also be exposed as an admin API endpoint (e.g. POST /admin/reindex)?
In-memory _documents dict is lost on restart — migrating to DocumentRegistry fixes this, but it's a broader change to the documents API. Confirm this is desired.
